2026-07-11 pre-extraction audit of packages/memory-schema: 2 high vulns, spec drifted behind code, schema mostly v5/SOTA-aligned with 3 divergences

## Observations

- SECURITY (verified vs source): HIGH-1 preflight._start_container runs 'docker compose -f <project_root>/docker-compose.yml up -d' (preflight.py:43) and _maybe_preflight runs auto_start=True on EVERY cli call (main.py:93,142) -> running any memoryschema subcommand in an untrusted cloned repo that ships a docker-compose.yml boots that stack = host compromise. Fix: never compose-up a file the package didn't generate; use 'docker start <name>' only, or gate auto_start opt-in. HIGH-2 hook-post-write.sh exports EVERY .env key unfiltered (line 86) before invoking python -> PYTHONPATH/LD_PRELOAD/NEO4J_URI poisoning from an untrusted repo's .env = code exec / silent index redirect. Fix: allowlist keys (NEO4J_*/VOYAGE_API_KEY/MEMORY_GENERATOR only), resolve python to absolute before loading .env. Both fire only when operating INSIDE an untrusted repo (low for a solo operator today) but MUST be fixed pre-extraction since the package becomes published+user-global.
- SECURITY (medium): (a) cross-project persistent prompt-injection - hook-indexed entities from any repo land project=NULL in the shared Neo4j (bolt://localhost:7687) and NULL is globally visible in recall (neo4j_store.py:414,439,738) -> a hostile repo's memory/*.md re-surfaces in your OTHER projects' recall; the [UNTRUSTED] marker exists only for corpus imports, not hook writes. (b) init's docker-compose.yml.tpl bakes the Neo4j password in plaintext x2 (the exact bug Helios hit in 310a42e) - template must use ${NEO4J_PASSWORD}. (c) store.jsonl locking is a no-op on Windows (fcntl only, store.py:89-94) -> concurrent hook+CLI writers silently lose updates on the deployed platform. LOW: 'remember <name>' path-joins name unvalidated (F3 misses '/' and '..' -> write .claude/rules/*.md outside memory/); hook-stop.sh splices .active_chain into systemMessage JSON unescaped. CLEAN: Cypher fully parameterized+allowlisted; no shell=True/eval/exec/pickle; XML is XXE-safe etree; secrets never in audit.jsonl/logs; writes atomic (mkstemp+os.replace).
- DOC ALIGNMENT - THE KEY RISK FOR EXTRACTION: docs/memory-system-specification.md (the DECLARED SSOT, 61KB) has DRIFTED BEHIND the code. The 2026-07-06/07 security-fix batch (40344f0, bf3635e, c190e0a - ~24 fixes) advanced the code but the spec still describes the OLD behavior in 2 MAJOR spots: spec 3.3 says reconcile refuses to run on malformed v5 (code: v4-only guard, a corrupt v5 is PRUNED = data loss); spec 4.3/4.6 describes the old supersede-before-index order that was deliberately reversed to fix key-stranding. Rebuilding from the spec would REINTRODUCE fixed bugs. Also stale: the widened relation-target charset, quote/frontmatter escaping, the /tmp->.memoryschema sentinel move, the (confirmed-stale) 'hook upgrade overwrites patches' warning, CHANGELOG missing the 07-06/07 batch.
